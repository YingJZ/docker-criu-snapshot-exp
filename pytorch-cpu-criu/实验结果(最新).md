## 6. 分析

### 6.1 裸机 CRIU 效果显著

裸机 CRIU restore 仅需 189ms，对比冷启动 1.853s 有 **6.8x 加速**。主要收益来自跳过了 `import torch` (834ms) 和 `import torchvision` (691ms) 两个最大的冷启动瓶颈。

### 6.2 Docker CRIU 层开销巨大

Docker CRIU 的 checkpoint/restore 耗时远超裸机 CRIU:
- Checkpoint: 9,175ms vs 274ms (**33.5x 更慢**)
- Restore: 6,812ms vs 189ms (**36.0x 更慢**)

开销来源:
1. **Docker 容器状态重建**: 网络 namespace、cgroup、文件系统挂载点等
2. **容器元数据序列化/反序列化**: Docker 需要额外处理容器配置
3. **网络 namespace 问题**: Docker 29.3.0 在 bridge 网络下 restore 失败，必须使用 `--network=host`

### 6.3 CRIU 数据完整性验证通过

所有方案中 CRIU restore 后推理结果与基线 **完全一致** (差异为 0.0)，证明:
- CPU tensor 数据在 CRIU restore 后完整保留
- 模型权重、RNG 状态均正确恢复
- Python 进程状态（变量、计数器、信号处理器）完整恢复

### 6.4 小模型场景结论

ResNet-50 (97.5MB) 属于小模型:
- **裸机 CRIU**: 有显著加速效果 (6.8x)
- **Docker CRIU**: 反而比冷启动更慢 (0.27x)
- 随模型规模增大，冷启动耗时线性增长，而 CRIU restore 耗时与 RSS 线性相关，大模型场景预期收益更大
